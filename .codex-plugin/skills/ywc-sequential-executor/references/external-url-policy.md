# External URL Policy (Spec Reference)

> Referenced from: SKILL.md → Pre-flight (item 5)
> Read this when: Pre-flight detects no existing `taskExecutor.externalSpecUrls` decision in `Codex approval settings or the project-local policy file`, OR when applying the policy in Step 1b's Spec Reference Loading.

## Why a separate decision is needed

Tasks generated by `ywc-task-generator` include a `Spec Reference` section that may list `Primary Sources`. By default, only project-relative paths are expected. External URLs require an explicit project-level decision because:

- **Network unpredictability** — Fetching external URLs mid-task can hang on network issues, rate limits, or auth walls — breaking the "don't stop mid-way" execution promise
- **Privacy risk** — External docs may contain content the user did not intend to expose to an LLM
- **Auth walls** — Some docs are behind SSO and cannot be fetched at all

A project-level, persisted decision eliminates per-task interruption.

## Procedure

Run this **once** at Pre-flight, before touching any task.

### 1. Check existing decision

Read `Codex approval settings or the project-local policy file`. Look for a `taskExecutor.externalSpecUrls` key:

```jsonc
{
  "permissions": { /* ... unchanged ... */ },
  "taskExecutor": {
    "externalSpecUrls": "deny",              // "deny" | "allow" | "allowlist"
    "externalSpecUrlAllowlist": [            // only used when policy is "allowlist"
      "github.com",
      "figma.com/file"
    ],
    "decidedAt": "2026-04-05"
  }
}
```

### 2. If the key exists, use it silently

Do **not** re-prompt the user. The whole point of persistence is to avoid re-asking.

### 3. If the key is missing, ask the user **once**

Use a concrete, answerable question:

> "Some tasks may list external URLs (Notion, Confluence, Figma, etc.) in their Spec Reference. How should I handle them?
> 1. **deny** (recommended) — ignore external URLs; use only project-relative paths and the Summary field
> 2. **allow** — fetch all external URLs when a task lists them
> 3. **allowlist** — I'll give you specific domains to allow (everything else is denied)
>
> I'll save your answer to `Codex approval settings or the project-local policy file` so I don't have to ask again."

### 4. Persist the decision

Write the decision back to `Codex approval settings or the project-local policy file` under the `taskExecutor` key. **Preserve every other key** in the file (especially `permissions`). If the file does not exist, create it with both `permissions: { "allow": [], "deny": [], "ask": [] }` and the `taskExecutor` key.

### 5. Enforce the policy in Step 1b (Spec Reference Loading)

When reading `Spec Reference > Primary Sources`:

| Policy | Behavior |
|---|---|
| **deny** | Ignore any URL starting with `http://` or `https://`. Rely on the Summary field and project-relative paths. Log which URLs were skipped so the user can see them in the completion report. |
| **allow** | Fetch every URL. Failures (network, 401, 404) are warnings, not errors; continue with what was retrieved. |
| **allowlist** | Fetch only URLs whose host+path prefix matches an entry in `externalSpecUrlAllowlist`. Skip the rest silently but log them. |

## Rationale for ask-once-and-persist

Stopping before every task to ask about URL handling would defeat the "don't stop mid-way" guarantee that makes range execution useful. A single project-level decision, stored in `settings.local.json` (which is gitignored), is the right granularity — it's per-machine, per-project, and survives across sessions.

The decision is also reversible: the user can edit `Codex approval settings or the project-local policy file` directly if they want to change policy without re-running pre-flight.
