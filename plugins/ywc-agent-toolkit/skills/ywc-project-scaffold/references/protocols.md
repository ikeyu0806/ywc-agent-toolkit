## GraphQL

### Additional Directory Structure

```
project-root/
├── src/  (or app/, internal/)
│   ├── graphql/                   # GraphQL layer
│   │   ├── schema/                # Type definitions
│   │   │   ├── user.graphql       # SDL-first approach
│   │   │   ├── item.graphql
│   │   │   └── schema.graphql     # Root schema (Query, Mutation, Subscription)
│   │   ├── resolvers/             # Resolver functions
│   │   │   ├── user.resolver.ts   # User query/mutation resolvers
│   │   │   ├── item.resolver.ts
│   │   │   └── index.ts           # Resolver aggregation
│   │   ├── directives/            # Custom directives (@auth, @cache, etc.)
│   │   │   └── auth.directive.ts
│   │   ├── scalars/               # Custom scalar types (DateTime, JSON, etc.)
│   │   │   └── date-time.scalar.ts
│   │   ├── dataloaders/           # DataLoader (prevents N+1 queries)
│   │   │   └── user.loader.ts
│   │   └── context.ts             # GraphQL context creation
│   ...
```

### Differences by Framework

| Framework                | Approach                                | Notes                                                      |
| ------------------------ | --------------------------------------- | ---------------------------------------------------------- |
| **NestJS**               | Code-first (`@Resolver`, `@ObjectType`) | `*.resolver.ts` serves as the Controller                   |
| **FastAPI** (Strawberry) | Code-first (Python decorator)           | Types and resolvers can be defined together in `schema.py` |
| **Rails** (graphql-ruby) | Code-first (`Types::`, `Mutations::`)   | `types/`, `mutations/` under `app/graphql/`                |
| **Go** (gqlgen)          | Schema-first (SDL → codegen)            | `graph/schema.graphqls` + `graph/resolver.go`              |
| **Next.js** (Apollo)     | Schema-first or Code-first              | `src/graphql/` or integrated into API routes               |

### NestJS + GraphQL Example

```
src/modules/users/
├── users.module.ts
├── users.resolver.ts              # @Resolver() - Defines Query/Mutation
├── users.service.ts
├── dto/
│   ├── create-user.input.ts       # @InputType()
│   └── user.type.ts               # @ObjectType()
└── entities/
    └── user.entity.ts
```

### FastAPI + Strawberry GraphQL Example

```
app/
├── graphql/
│   ├── schema.py                  # Strawberry schema definition
│   ├── types/
│   │   ├── user_type.py           # @strawberry.type
│   │   └── item_type.py
│   ├── mutations/
│   │   └── user_mutations.py
│   ├── queries/
│   │   └── user_queries.py
│   └── dataloaders/
│       └── user_loader.py
├── services/                      # Uses existing service layer as-is
└── repositories/
```

### Go + gqlgen Example

```
internal/
├── graph/
│   ├── schema.graphqls            # SDL schema definition
│   ├── generated.go               # gqlgen codegen output (do not modify)
│   ├── resolver.go                # Resolver struct
│   ├── user.resolvers.go          # User-related resolver implementations
│   └── model/
│       └── models_gen.go          # Generated models
├── service/                       # Existing service layer
└── repository/
```

## gRPC

### Additional Directory Structure

```
project-root/
├── proto/                         # Protobuf definitions (root level)
│   ├── user/
│   │   └── v1/
│   │       └── user.proto         # Service + Message definitions
│   ├── item/
│   │   └── v1/
│   │       └── item.proto
│   └── common/
│       └── v1/
│           └── pagination.proto   # Common message types
├── gen/                           # Protobuf codegen output
│   ├── go/                        # Go generated code
│   │   └── user/v1/
│   │       ├── user.pb.go
│   │       └── user_grpc.pb.go
│   ├── python/                    # Python generated code
│   │   └── user/v1/
│   │       ├── user_pb2.py
│   │       └── user_pb2_grpc.py
│   └── ts/                        # TypeScript generated code
├── src/ (or internal/)
│   ├── grpc/
│   │   ├── server.ts              # gRPC server setup
│   │   ├── handlers/              # gRPC service implementations
│   │   │   └── user_handler.ts
│   │   └── interceptors/          # gRPC interceptors (middleware)
│   │       ├── auth.interceptor.ts
│   │       └── logging.interceptor.ts
│   ...
├── buf.yaml                       # Buf configuration
├── buf.gen.yaml                   # Buf codegen config
└── Makefile                       # proto compile commands
```

### Key Points

- `proto/`: Language-agnostic Protobuf definitions. Versioning (`v1/`) is required
- `gen/`: Codegen output. Whether to add to `.gitignore` or commit is a team policy decision
- Buf is recommended: linting, breaking change detection, codegen
- Use `grpc-gateway` when providing both gRPC and REST simultaneously

### Go + gRPC Example

```
internal/
├── handler/
│   ├── http/                      # REST handlers (existing)
│   │   └── user_handler.go
│   └── grpc/                      # gRPC handlers
│       └── user_handler.go        # Implements generated interface
├── service/                       # Shared business logic
│   └── user_service.go
└── repository/
```

### Python + gRPC Example

```
app/
├── grpc/
│   ├── server.py                  # gRPC server bootstrap
│   └── servicers/                 # gRPC service implementations
│       └── user_servicer.py
├── services/                      # Shared business logic
└── repositories/
```

## WebSocket

### Additional Directory Structure

```
project-root/
├── src/ (or app/)
│   ├── websocket/                 # WebSocket layer
│   │   ├── server.ts              # WS server setup
│   │   ├── handlers/              # Event handlers
│   │   │   ├── connection.handler.ts
│   │   │   ├── chat.handler.ts
│   │   │   └── notification.handler.ts
│   │   ├── events/                # Event type definitions
│   │   │   └── events.ts          # Event name constants + payload types
│   │   ├── rooms/                 # Room/Channel management
│   │   │   └── room.manager.ts
│   │   └── middleware/            # WS middleware (auth, etc.)
│   │       └── ws-auth.middleware.ts
│   ...
```

### Differences by Framework

| Framework   | WebSocket Support                   | Notes                                |
| ----------- | ----------------------------------- | ------------------------------------ |
| **NestJS**  | `@nestjs/websockets` + Socket.IO/WS | Uses `@WebSocketGateway()` decorator |
| **FastAPI** | Built-in WebSocket support          | `@app.websocket()` route decorator   |
| **Rails**   | Action Cable                        | `app/channels/` directory            |
| **Next.js** | API Route + WS library              | May require a custom server          |
| **Go**      | gorilla/websocket, nhooyr/websocket | Based on standard library            |

### NestJS + WebSocket (Gateway)

```
src/modules/chat/
├── chat.module.ts
├── chat.gateway.ts                # @WebSocketGateway()
├── chat.service.ts
└── dto/
    ├── send-message.dto.ts
    └── join-room.dto.ts
```

### FastAPI + WebSocket

```
app/
├── websocket/
│   ├── connection_manager.py      # WebSocket connection pool management
│   ├── routes.py                  # WebSocket endpoint definitions
│   └── handlers/
│       └── chat_handler.py
```

### Rails + Action Cable

```
app/
├── channels/
│   ├── application_cable/
│   │   ├── connection.rb          # Auth, connection setup
│   │   └── channel.rb             # Base channel
│   ├── chat_channel.rb            # Feature channel
│   └── notification_channel.rb
```

## Message Queue

### Additional Directory Structure

Asynchronous messaging using Kafka, RabbitMQ, Redis Pub/Sub, etc.

```
project-root/
├── src/ (or app/, internal/)
│   ├── messaging/                 # Message queue layer
│   │   ├── producers/             # Message producers (publishers)
│   │   │   ├── user_event.producer.ts
│   │   │   └── order_event.producer.ts
│   │   ├── consumers/             # Message consumers (subscribers)
│   │   │   ├── user_created.consumer.ts
│   │   │   └── order_placed.consumer.ts
│   │   ├── schemas/               # Message schema/contract
│   │   │   ├── user_event.schema.ts
│   │   │   └── order_event.schema.ts
│   │   └── config/
│   │       └── queue.config.ts    # Queue/Topic configuration
│   ...
```

### Differences by Framework

| Framework   | Common Choice                         | Notes                                         |
| ----------- | ------------------------------------- | --------------------------------------------- |
| **NestJS**  | `@nestjs/microservices`               | Integrated support for Kafka, RabbitMQ, Redis |
| **FastAPI** | Celery, arq, or custom implementation | Separated from background tasks               |
| **Rails**   | Sidekiq, Karafka (Kafka)              | `app/jobs/` + `app/consumers/`                |
| **Go**      | Sarama (Kafka), amqp (RabbitMQ)       | `internal/messaging/` or `internal/event/`    |

### Go + Message Queue

```
internal/
├── event/
│   ├── publisher/
│   │   └── kafka_publisher.go
│   ├── subscriber/
│   │   └── user_event_subscriber.go
│   └── schema/
│       └── user_event.go          # Event struct definition
```

## Composite Protocol

Strategy for integrating structures when supporting multiple protocols simultaneously:

### REST + GraphQL

Add GraphQL as a separate endpoint. Maintain the existing REST structure.

```
src/
├── api/                           # REST API (existing)
│   └── v1/
├── graphql/                       # GraphQL endpoint added
│   ├── schema/
│   └── resolvers/
├── services/                      # Shared (used by both REST and GraphQL)
└── repositories/                  # Shared
```

### REST + gRPC

Separate only the transport layer; share business logic.

```
src/
├── transport/                     # Transport layer separation
│   ├── http/                      # REST handlers
│   └── grpc/                      # gRPC handlers
├── services/                      # Shared
└── repositories/                  # Shared
```

### REST + WebSocket + Message Queue

```
src/
├── api/                           # REST
├── websocket/                     # Real-time
├── messaging/                     # Async
├── services/                      # Shared (called from all transports)
└── repositories/
```

**Integration Principles:**

- Separate only the Transport/Delivery layer per protocol
- Service, Repository, and Domain layers are always shared
- Adding a new protocol should not modify the existing structure (Open-Closed Principle)
